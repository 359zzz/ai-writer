## 1）北极星（不要轻易修改）

### 1.0 当前阶段摘要
- 项目当前已从“基础多 Agent 小说写作 MVP”演进为：**本地单用户的创作 / 文章续写 / 长书续写 / 图谱回放 / Agent 可观测协作平台**。
- 顶层导航以 `创作 / 续写 / 图谱 / Agent 协作 / 设置` 为稳定界面结构。
- 当前推荐能力闭环不再只是“写一章”，而是：项目设定 → 知识库 / 大纲 → 写作 / 续写 → trace / 图谱排障 → 导出。
- 当前工程重点已经明确扩展到：长书续写、章节结构化、图谱生成、长任务可恢复、代理网关稳定性。

### 1.1 产品目标
构建一个本地、单用户、Web 端的 **多 Agent 协作小说工作台**，具备以下核心能力：
- 创作：项目管理、背景设定、本地知识库、大纲编辑（文本块 / 导图）、章节写作、批量生成。
- 文章续写：输入已有文本，提取 `StoryState` 后继续写作。
- 书籍续写：围绕长书 / 长文建立章节级结构、摘要、状态与续写工作流。
- 图谱：回放大纲图、书籍结构图、人物关系图，以及运行过程的聚合图谱。
- Agent 协作：展示时间线、工具调用、产物与错误，支持排障与回放。
- 设置与导出：本地配置模型 / 密钥 / 项目参数，并导出 `DOCX / EPUB / PDF`。

### 1.2 创作与续写原则
- 用户定义的故事设定（角色、大纲、世界观、风格、篇幅、章节等）默认都是可选项。
- 弱依赖模式下，缺失设定可由 LLM 创造性补全，但不得覆盖用户已填写内容。
- 强依赖模式下，世界观 / 正典事实必须来自本地知识库、用户配置、既有正文或书籍状态；缺失时应标记待确认，而不是擅自编造。
- 创作链路与续写链路都应支持“边运行、边落库、边可见”，避免长任务完成前完全不可观察。

### 1.3 长书续写工作流
- 长书续写是当前平台的核心方向之一，推荐链路为：
  - 上传本地书籍源
  - 规则优先进行章节识别与章节索引
  - 允许用户手动微调章节边界 / 标题 / 编号
  - 按章总结入本地知识库（`book_summarize`）
  - 编译书籍状态（`book_compile`）
  - 生成书籍结构 / 人物 / 关系图谱
  - 进入书籍续写工作台继续生成后续章节（`book_continue`）
- 章节识别应优先依赖本地规则与结构化索引，而不是默认依赖 LLM 做分章。
- 书籍续写链路必须兼顾：连续性、可恢复性、可观察性、对代理网关波动的容错。

### 1.4 知识库与联网检索
- 本地知识库是项目内部的长期记忆层；在强依赖模式下，它是规范事实来源。
- 联网检索是外部研究工具，只提供临时参考；结果不得自动写入本地知识库。
- 已生成章节、书籍总结、书籍状态、关系图谱等产物可以本地落库，并通过标签 / 元数据参与后续检索与图谱关联。
- 对长书场景，应优先使用知识库元数据与结构化标签，而不是在前端反复加载大段正文。

### 1.5 多 Agent 编排与可观测性
- 使用 **Supervisor / Orchestrator** 模式，由 Director 统一调度各类 Agent 与工具。
- 通用创作链路中的关键角色包括：`ConfigAutofill`、`Extractor`、`Outliner`、`Writer`、`Editor`、`LoreKeeper`。
- 长书链路中的关键步骤 / 角色包括：`book_summarize`、`book_compile`、`book_continue`，以及围绕图谱生成的关系 / 人物抽取流程。
- 在 PackyAPI 的文章续写链路中，应优先保证“真实产物稳定产出”而不是让单一 provider 独占全部步骤：
  - `ConfigAutofill` / `Extractor` / `Outliner` 在 Gemini + Packy 组合下，允许走稳定的 OpenAI-compatible 结构化通道。
  - `Writer` 必须优先使用用户在界面中实际选中的模型；只有在模型不可用、空响应、明显过短或代理网关异常时，才允许透明 fallback。
  - `Editor` 在 Gemini + Packy 组合下，允许走稳定的 OpenAI-compatible 通道，以提高章节产物稳定性与可诊断性。
- 每次 `run` 都必须产生可持久化的 `trace` 事件流，记录开始 / 结束、工具调用、中间输出、错误与最终产物。
- 前端既要支持 SSE 实时显示，也要支持基于 `run` 与事件序列的轮询恢复，保证刷新页面后仍可恢复进度。

### 1.6 架构（Monorepo）
- `apps/api`：FastAPI 后端，负责 SSE、轮询接口、编排、存储、导出、工具路由。
- `apps/web`：Next.js 前端，负责创作 / 续写 / 图谱 / Agent 协作 / 设置界面。
- `scripts`：Windows PowerShell 本地开发脚本，主入口为 `scripts/dev.ps1`。
- 存储：
  - SQLite：项目、章节、设置、`run`、`trace`、知识库切片等持久数据
  - 本地文件系统：导出文件、continue source、章节索引、本地密钥文件等
- 长任务相关状态应优先通过数据库与本地文件持久化，而不是仅保存在内存中。

### 1.7 约束与安全
- 严禁提交密钥；`api.txt`、`.env*`、本地 secrets 文件必须保持 git 忽略。
- 设置界面绝不能显示完整 API Key，也不得在日志 / trace 中泄漏密钥。
- 在任何调用真实 LLM API 的自动化测试中，输出必须保持短小（不超过约 500 中文字符 / 500 英文词）。
- 对 OpenAI-compatible / Gemini / 代理网关的兼容性处理，要优先追求稳定与可诊断，不要为了“探测更多可能性”而制造过多噪声请求。

### 1.8 长期维护热点
- `apps/api/ai_writer_api/routers/runs.py`：运行编排、长书链路、SSE、错误处理高度集中，是后端首要热点。
- `apps/web/src/app/page.tsx`：前端状态与工作区编排高度集中，是前端首要热点。
- `apps/api/ai_writer_api/llm.py`：模型兼容、重试、fallback、PackyAPI 策略集中，改动必须谨慎。
- `apps/api/ai_writer_api/tools/chapter_index.py`：章节识别规则密集，任何调整都应由回归测试兜底。
- `apps/web/e2e/continue-acceptance.spec.ts`：文章续写的真实界面验收脚本；模型兼容或续写编排改动后必须优先回归。
- 当这些热点的职责明显变化时，必须同步更新本文件与 `HANDOFF.md`。
